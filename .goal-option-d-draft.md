Build `pdfplumber.js` into a stable browser-capable JS alternative to Python `pdfplumber`.

Stable means: the public extraction API is documented; the same API passes in Node and real browsers; users can extract PDF data without knowing which internal engine is used. The supported public API covers metadata, pages, boxes, text, chars, words, search, crop/filter, dedupe, vectors, colors, annotations, marked content, image metadata, and table inputs. The browser package is ESM and accepts `ArrayBuffer`, `Blob`, and `URL`.

The implementation target is native pdfminer-style parsing/extraction. pdf.js may remain only for named tested roles. MuPDF may be used as a differential check, not as the primary oracle. Python `pdfplumber` and `pdfminer.six` are the primary behavior oracles. No mismatch may be fixed by output matching. Each mismatch must produce a general rule, upstream reference, targeted test, and real-corpus validation; or be classified as upstream-compatible limitation, backend gap, unsupported feature, intentional difference, duplicate coverage, or excluded behavior.

The upstream contract consists of every item in:

- `pdfplumber-python/tests`
- `pdfminer-six/tests`
- `pdfjs/test/test_manifest.json`
- `pdfjs/test/unit`
- `pdfjs/test/font`
- `pdfjs/test/integration`

Create a committed dashboard covering every upstream item. Each row must include: source, asserted behavior, scope, subsystem, JS test or reason, status, and rationale. No item may be omitted. Allowed scopes are: `public-api`, `runtime-adaptation`, `native-engine`, `pdfjs-capability`, `robustness-corpus`, `duplicate`, and `excluded`.

The upstream contract passes only when:

- All `public-api` and `runtime-adaptation` items pass in adapted JS tests.
- All `native-engine` items for default native subsystems pass in JS tests.
- All `pdfjs-capability` items for capabilities used or exposed by `pdfplumber.js` pass in JS tests.
- All `robustness-corpus` items either pass or raise the documented stable error.
- `duplicate` items reference the covering test.
- `excluded` items are limited to behavior not exposed by this library, such as visual rendering, editor/viewer UI, Python-only behavior, CLI-only behavior, or PIL/Jupyter debug helpers.

Browser compatibility passes only when Playwright runs the built browser ESM package in Chromium, Firefox, and WebKit, using `ArrayBuffer`, `Blob`, and `URL` inputs, and verifies the same public extraction behavior as Node against Python goldens or validated Node results.

A real-document cycle measures generalization beyond upstream fixtures. Each cycle adds 120 counted real-document page cases using the existing parity-cycle manifest, golden, and shard organization: 100 working cases and 20 holdout cases. Counted cases must come from real PDFs or established upstream corpora, using selected pages or short ranges. Generated PDFs are diagnostics only.

For every counted case, record origin, terms, SHA-256, size, selected pages, source/corpus, metadata, categories, and why the page matters. Working and holdout cases must use different source documents. Expected outputs must come from Python `pdfplumber`/`pdfminer.six` whenever possible and must assert meaningful structured behavior, not only text.

During a cycle, development may use only the 100 working cases and focused diagnostics. The 20 holdout cases remain unopened until 100/100 working cases pass. A cycle succeeds only if holdout has at most 1 failure out of 20. A failed cycle must classify failures and resets the consecutive-success count.

The goal is complete only when the upstream contract passes, browser compatibility passes, native subsystems for text, words, geometry, boxes, paths, annotations, marked content, image metadata, and table inputs are default, pdf.js capability regressions are classified by the dashboard, MuPDF deltas are classified when MuPDF is used, and 3 real-document cycles succeed consecutively.
