**Goal**

Build `pdfplumber.js` into a browser-capable JS alternative to Python `pdfplumber`, validated by **3 consecutive successful real-document parity cycles**.

Work one cycle at a time. Each cycle adds **100 working** real-document cases and **20 untouched holdout** cases. During a cycle, develop only against the working set and focused manual checks. A cycle may reach holdout validation only after **100/100 working cases pass**. It succeeds only if holdout has **at most 1 failure out of 20**. The goal is reached only after **3 successful cycles in a row**. If a cycle fails, learn and continue, but the consecutive-success count resets.

Counted cases must come from real PDFs or established upstream corpora, using selected pages or short page ranges. Generated PDFs are allowed only as non-counted diagnostics after a real document reveals a behavior worth isolating.

Each cycle should deliberately expand real-world coverage. Prefer diversity of document domains, producers, languages, layouts, and PDF generators over many easy pages from the same kind of file. When existing fixtures are not enough, download stable public or libre PDFs and record their provenance. Include difficult real PDFs when useful, including malformed, encrypted, repaired, or Python-error cases, as long as Python behavior is observable and selected pages stay fast.

Before choosing the next cycle, review what previous cycles, holdouts, and existing tests already exposed. The new working set should cover missing behavior families, weakly generalized areas, and failures revealed by prior holdouts. Also rerun the relevant previous tests before declaring progress, so new fixes do not break behavior that was already working.

Every counted case must be reproducible and traceable: record origin, license/terms note, SHA-256, file size, selected pages, source/corpus, useful metadata, categories, and why the selected page matters. Follow the existing manifest, golden, and shard organization used by the parity-cycle scripts.

Expected outputs must come from Python `pdfplumber`/`pdfminer.six` whenever possible. Tests should compare structured behavior, not just text. The snapshot can be compact, but it must still detect meaningful parity regressions in geometry, text flow, chars/words, objects, graphics, colors, images, annotations, tags, search, or tables when those features matter.

The harness is part of the product. Large outputs may be summarized, but summaries must include counts, representative beginning/middle/end samples, and a whole-output fingerprint or equivalent check that is actually asserted. A green cycle does not count if assertions are too weak to detect real differences. Summaries, numeric tolerances, skipped cases, and holdout thresholds must be asserted deliberately rather than interpreted by hand.

When a working case or holdout exposes a bug, first understand the behavior before changing JS. Read the relevant upstream source and tests, especially Python `pdfplumber`, `pdfminer.six`, and dependencies when they define the oracle. Use any necessary tools or small Python/Node terminal experiments to confirm how the behavior works. Then implement the general rule in JS and add focused tests plus real-document coverage.

Holdouts must remain independent from same-cycle working documents and untouched until validation. A holdout failure is feedback about what did not generalize; classify it and feed that behavior into the next cycle instead of treating it as a one-off.

Keep iteration fast and browser-oriented with selected pages, sharding, and explicit timeouts. If a document is too slow, reduce it to behaviorally relevant selected pages instead of replacing it with an easier case. Scripts must not silently skip cases or hide failing shards. After each coherent behavior-family improvement, commit implementation, tests, provenance, regenerated goldens, validation notes, and any upstream or terminal behavior that guided the fix together.
